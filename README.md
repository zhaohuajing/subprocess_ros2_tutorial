# Using Python `subprocess` to Build ROS 2 Wrappers for Dockerized Perception and Grasp-Planning Modules

## Overview

This tutorial shows how to use Python `subprocess` to build ROS 2 wrappers around third-party perception and grasp-planning modules that must run in external environments such as Docker containers or separate conda environments.

The main motivating example is Contact-GraspNet, wrapped as a ROS 2 service-client interface. In this design:

- ROS 2 runs on the host machine, for example with Python 3.12 and CUDA 12.2.
- Contact-GraspNet inference runs inside a Docker container, for example with Python 3.9 and CUDA 11.8.
- The ROS 2 server uses `subprocess.run()` to invoke Docker commands, execute inference, capture structured output, and return grasp results back to ROS 2.

The same design pattern can also be extended to other modules, such as UnseenObjectClustering, segmentation pipelines, or legacy research code that is hard to install directly into a host ROS 2 environment.

## Learning goals

By the end of this tutorial, you should be able to:

- understand what `subprocess.run()` does
- use `subprocess` to launch shell commands from Python
- capture standard output and standard error
- use `subprocess` to call commands inside a Docker container
- understand why this pattern is useful for ROS 2 wrappers
- follow the structure of a ROS 2 server that uses `subprocess` to call Contact-GraspNet inference inside Docker

## Why use `subprocess` here

Many open-source robotics modules are powerful but difficult to integrate cleanly into a modern ROS 2 stack because they may require:

- different Python versions
- older CUDA or TensorFlow versions
- legacy dependencies
- custom compile steps
- isolated conda or Docker environments

Instead of forcing all dependencies into the same host environment, we can let the host ROS 2 node act as an orchestrator and use `subprocess` to launch the external command in the correct environment.

In other words:

- the host ROS 2 node handles communication
- Docker or conda handles the external runtime
- `subprocess` acts as the bridge between them

## Part 1. Minimal local example: create and write a file with `subprocess`

The goal of this example is to show that a Python script can launch a shell command that creates a file and writes text into it.

### Example script

Use the provided script:

`./scripts/part1_create_file.py`

```python
#!/usr/bin/env python3
import subprocess
from pathlib import Path

project_root = Path(__file__).resolve().parent.parent
output_dir = project_root / "outputs"
output_dir.mkdir(exist_ok=True)
output_file = output_dir / "demo_output.txt"

cmd = [
    "bash",
    "-lc",
    f"echo 'Hello from subprocess' > '{output_file}'"
]

result = subprocess.run(cmd, capture_output=True, text=True)

print("Return code:", result.returncode)
print("STDOUT:", result.stdout)
print("STDERR:", result.stderr)
print("Created file:", output_file)
```

### How it works

The `cmd` list defines the external command that Python will execute.

- `"bash"` launches a Bash shell
- `"-lc"` tells Bash to run the following command string
- `f"echo 'Hello from subprocess' > '{output_file}'"` writes text into a file under the `outputs` folder

The command part is equivalent to running the following command in a terminal:

```bash
echo 'Hello from subprocess' > outputs/demo_output.txt
```

You may check this manually by running the following commands in a terminal:

```bash
rm -rf outputs/demo_output.txt
echo 'Hello from subprocess' > outputs/demo_output.txt
cat outputs/demo_output.txt
```

### Suggested terminal commands

Run the Python example from the tutorial root folder:

```bash
cd subprocess_ros2_tutorial_v2
python scripts/part1_create_file.py
cat outputs/demo_output.txt
```

### Expected result

You should see a file named `demo_output.txt` created in the `outputs` directory, containing:

```text
Hello from subprocess
```

### Slightly extended version

This version writes two lines and prints the file content back to standard output.

Use:

`./scripts/part1_create_file_extended.py`

```python
#!/usr/bin/env python3
import subprocess
from pathlib import Path

project_root = Path(__file__).resolve().parent.parent
output_dir = project_root / "outputs"
output_dir.mkdir(exist_ok=True)
output_file = output_dir / "demo_output.txt"

cmd = [
    "bash",
    "-lc",
    f"echo 'Line 1' > '{output_file}' && echo 'Line 2' >> '{output_file}' && cat '{output_file}'"
]

result = subprocess.run(cmd, capture_output=True, text=True)

print("Return code:", result.returncode)
print("STDOUT:")
print(result.stdout)
print("STDERR:")
print(result.stderr)
print("Created file:", output_file)
```

The command part is equivalent to running the following terminal commands:

```bash
echo 'Line 1' > outputs/demo_output.txt
echo 'Line 2' >> outputs/demo_output.txt
cat outputs/demo_output.txt
```

### Error example

## Part 2. Returning structured output with JSON

When integrating external tools, it is often better to return structured output rather than plain text. JSON is a convenient format because it can be parsed directly in Python.

### Example script

Use:

`./scripts/part2_json_example.py`

```python
#!/usr/bin/env python3
import subprocess
import json

cmd = [
    "python3",
    "-c",
    (
        "import json; "
        "print(json.dumps({'status': 'ok', 'message': 'hello', 'values': [1, 2, 3]}))"
    )
]

result = subprocess.run(cmd, capture_output=True, text=True)

if result.returncode != 0:
    raise RuntimeError(result.stderr)

data = json.loads(result.stdout)
print("Parsed JSON:", data)
```

### Equivalent terminal command

The subprocess command is equivalent to running:

```bash
python -c "import json; print(json.dumps({'status': 'ok', 'message': 'hello', 'values': [1, 2, 3]}))"
```

### Suggested terminal commands

```bash
cd subprocess_ros2_tutorial_v2
python scripts/part2_json_example.py
```

### Why this matters later

In the Contact-GraspNet wrapper, the ROS 2 server runs inference externally, captures the command output, and extracts JSON containing values such as:

- `pred_grasps_cam`
- `scores`
- `contact_pts`

So this small example introduces the same concept on a simpler scale.

## Part 3. Running a command inside Docker from the host

Now we use the same `subprocess` idea, but instead of running a local shell command, we execute a command inside a running Docker container.

### Goal

From the host machine, call a command inside Docker that creates a file and prints it.

### First create a small demo container

Before running the Python subprocess example, create a sample Docker container that we can safely test with.

Run:

```bash
docker run -itd --name demo_container ubuntu:22.04 bash
docker exec demo_container bash -lc "apt-get update && apt-get install -y python3"
```

After creating the image and container, it is helpful to inspect what is available.

For example:

```bash
docker images
docker ps
```

- `docker images` shows the images available on your system
- `docker ps` shows the running containers
- you should see a container named `demo_container`

Once this is ready, we can move on to the Python script that uses `subprocess` to call Docker commands from the host.

### Example script

Use:

`./scripts/part3_docker_example.py`

```python
#!/usr/bin/env python3
import subprocess

container_name = "demo_container"

cmd = [
    "docker", "exec", container_name,
    "bash", "-lc",
    "echo 'Hello from inside Docker' > /tmp/docker_demo.txt && cat /tmp/docker_demo.txt"
]

result = subprocess.run(cmd, capture_output=True, text=True)

print("Return code:", result.returncode)
print("STDOUT:")
print(result.stdout)
print("STDERR:")
print(result.stderr)
```

### Equivalent terminal command

This subprocess command is equivalent to running the following command directly in a terminal:

```bash
docker exec demo_container bash -lc "echo 'Hello from inside Docker' > /tmp/docker_demo.txt && cat /tmp/docker_demo.txt"
```

### Suggested setup and test commands

After the container is created, run the example:

```bash
cd subprocess_ros2_tutorial_v2
python scripts/part3_docker_example.py
docker exec demo_container bash -lc "cat /tmp/docker_demo.txt"
```

### Why `bash -lc` is useful

Using `bash -lc` makes it easier to:

- chain commands with `&&`
- change directories
- call environment setup commands
- run `conda run ...`
- keep the entire Docker-side logic in one executable command string

This is especially useful for robotics wrappers where the external code needs to run in a very specific directory or environment.

## Part 4. A toy ROS 2 service server using `subprocess`

Before moving to Contact-GraspNet, it is helpful to build a minimal ROS 2 service server that uses `subprocess` for a very simple task.

### Idea

The server receives a request and then uses `subprocess` to create a text file. The server returns a success message to the client.

This keeps the ROS 2 structure simple while preserving the same subprocess pattern.

### Example server script

This tutorial bundle includes a runnable toy ROS 2 server:

`./scripts/part4_toy_ros2_subprocess_server.py`

Its service callback uses `subprocess` to run a shell command that creates `/tmp/ros2_demo.txt`. The uploaded example implementation is the basis for this part of the tutorial.

The callback is:

```python
def callback(self, request, response):
    cmd = [
        "bash",
        "-lc",
        "echo 'Created by ROS 2 subprocess server' > /tmp/ros2_demo.txt"
    ]

    result = subprocess.run(cmd, capture_output=True, text=True)

    if result.returncode != 0:
        response.success = False
        response.message = result.stderr
        return response

    response.success = True
    response.message = "Created /tmp/ros2_demo.txt successfully."
    return response
```

### Equivalent terminal command

The subprocess command above is equivalent to running:

```bash
echo 'Created by ROS 2 subprocess server' > /tmp/ros2_demo.txt
```

### What this teaches

Even though this example is simple, the pattern is identical to a real wrapper:

- ROS 2 callback receives a request
- Python launches an external command
- command output or status is checked
- ROS 2 returns a structured response

### Example ROS 2 workspace setup

## Part 5. Contact-GraspNet case study

We now move to the full wrapper idea: a ROS 2 service server that runs Contact-GraspNet inference inside a Docker container using `subprocess`.

### Repository references

For a fuller implementation, users can refer to the following repositories:

- Contact-GraspNet ROS 2 wrapper repository:
  `https://github.com/zhaohuajing/contact_graspnet_ros2/tree/ros2_server`
- A separate Docker repository for the Contact-GraspNet environment:
  `https://github.com/zhaohuajing/contact_graspnet_docker`

A straightforward way to set up the Docker image is to pull the prebuilt image from Docker Hub:

```bash
docker pull zhaohuajing/cuda118:contact_graspnet
```

So, in addition to the explanation in this tutorial, users can also reference the repositories and their README files for a ROS 2 service server that runs Contact-GraspNet inference inside a Docker container using `subprocess`.

### Core design

The server does not execute Contact-GraspNet directly inside the host ROS 2 Python environment.

Instead, the server:

1. receives a scene name from a ROS 2 client
2. builds a Docker command
3. uses `subprocess.run()` to execute inference inside the container
4. captures stdout
5. extracts JSON results
6. converts the results into ROS messages
7. sends them back in the service response

### Why this is useful

This allows you to:

- keep ROS 2 on the host system
- preserve a stable, isolated Contact-GraspNet environment in Docker
- avoid polluting the host ROS 2 environment with incompatible dependencies
- reuse the same wrapper pattern for other perception and grasp modules

## Walkthrough of the subprocess block in the Contact-GraspNet server

The following section summarizes the key subprocess logic from the Contact-GraspNet server implementation, based on the uploaded server example.

### A representative function

The bundle includes:

`./scripts/part5_contact_graspnet_subprocess_example.py`

```python
def run_inference_in_docker(self, scene_name) -> str:
    container_name = "contact_graspnet_container"
    np_path = f"test_data/{scene_name}.npy"

    compiled_lib = (
        "/root/graspnet_ws/src/contact_graspnet_ros2/contact_graspnet/"
        "pointnet2/tf_ops/sampling/tf_sampling_so.so"
    )

    compile_cmd = (
        f"if [ ! -f {compiled_lib} ]; then "
        f"cd /root/graspnet_ws/src/contact_graspnet_ros2/contact_graspnet && "
        f"conda run -n contact-graspnet bash compile_pointnet_tfops.sh; "
        f"fi"
    )

    inference_cmd = (
        "cd /root/graspnet_ws/src/contact_graspnet_ros2/contact_graspnet && "
        f"conda run -n contact-graspnet python contact_graspnet/inference.py "
        f"--np_path={np_path} --local_regions --filter_grasps"
    )

    cmd = [
        "docker", "exec", container_name,
        "bash", "-lc", f"{compile_cmd} && {inference_cmd}"
    ]

    result = subprocess.run(cmd, capture_output=True, text=True)
    if result.returncode != 0:
        raise RuntimeError(f"Inference failed: {result.stderr}")

    return result.stdout
```

### Equivalent terminal command

Conceptually, the `cmd` list above is equivalent to running a command such as the following in a terminal:

```bash
docker exec contact_graspnet_container bash -lc "if [ ! -f /root/graspnet_ws/src/contact_graspnet_ros2/contact_graspnet/pointnet2/tf_ops/sampling/tf_sampling_so.so ]; then cd /root/graspnet_ws/src/contact_graspnet_ros2/contact_graspnet && conda run -n contact-graspnet bash compile_pointnet_tfops.sh; fi && cd /root/graspnet_ws/src/contact_graspnet_ros2/contact_graspnet && conda run -n contact-graspnet python contact_graspnet/inference.py --np_path=test_data/<scene_name>.npy --local_regions --filter_grasps"
```

This is one of the most important ideas in the tutorial: `subprocess` is not doing anything mysterious. It is simply letting a Python script run the same kinds of commands that you might otherwise type manually into a terminal.

### What each part is doing

`container_name = "contact_graspnet_container"`
- selects the running Docker container that holds the Contact-GraspNet environment

`np_path = f"test_data/{scene_name}.npy"`
- identifies the input scene file inside the containerized workflow

`compiled_lib = ...`
- points to the shared object file expected after PointNet TensorFlow ops are compiled

`compile_cmd = ...`
- compiles TensorFlow ops only if the compiled shared library is not already present

`inference_cmd = ...`
- runs Contact-GraspNet inference with the target scene input

`cmd = [...]`
- packages the complete Docker-side execution into one subprocess command

`result = subprocess.run(cmd, capture_output=True, text=True)`
- executes the command and captures standard output and standard error as Python strings

### Why conditional compilation is useful

The wrapper checks whether the compiled library exists before running inference. This prevents unnecessary rebuilds and makes repeated server calls more efficient.

### Why JSON output is useful

A robust wrapper should return structured output rather than relying on loosely formatted log text.

In the full Contact-GraspNet server, the server tries to extract JSON between markers such as:

- `<<<BEGIN_JSON>>>`
- `<<<END_JSON>>>`

This makes the parsing more reliable when there are other logs printed to standard output.

## Suggested execution workflow for the full Contact-GraspNet wrapper

A typical workflow looks like this:

1. Prepare or mount the scene `.npy` input
2. Start the Contact-GraspNet Docker container
3. Launch the ROS 2 server on the host
4. Call the ROS 2 service client with a scene name
5. Let the server invoke Docker inference through `subprocess`
6. Parse the returned JSON
7. Convert the result into ROS messages and return them to the client

### Typical server-side steps

On the host:

```bash
source /opt/ros/<your_ros2_distro>/setup.bash
source install/setup.bash
ros2 run contact_graspnet_ros2 grasp_server
```

From another terminal:

```bash
source /opt/ros/<your_ros2_distro>/setup.bash
source install/setup.bash
ros2 service call /get_grasps contact_graspnet_ros2/srv/GetGrasps "{scene_name: 'sample_scene'}"
```

The exact package and service names may vary in your workspace.

## Final takeaways

The main lesson of this tutorial is that Python `subprocess` can serve as a practical bridge between:

- a modern ROS 2 host environment
- an isolated Docker or conda runtime for legacy or dependency-heavy models
- a clean ROS 2 service interface for the rest of the system

The same pattern can be reused for:

- Contact-GraspNet
- UnseenObjectClustering
- segmentation pipelines
- older perception or grasp-planning repositories
- research code that is difficult to install directly into the host ROS 2 environment

In other words, `subprocess` helps turn environment-fragile research code into ROS 2-accessible modules.
